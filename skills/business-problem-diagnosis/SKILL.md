# 企业问题诊断 Skill

> Public Version v1.0  
> 适用场景：帮助中小企业老板和管理者判断一个问题是否值得投入资源，识别真正的 A 类问题，并形成下一步行动建议。

---

## 1. Skill 定位

本 Skill 用于帮助企业主、管理者、咨询顾问和 AI 工作流设计者，对一个模糊的经营问题、管理问题或项目卡点进行初步诊断。

它的核心不是直接给答案，而是先帮助用户判断：

> 这个问题到底是不是值得现在投入精力、时间和资源处理的关键问题？

本 Skill 将问题拆解为：

- 问题事实
- 问题类型
- 影响范围
- 紧急程度
- 可控程度
- 资源投入价值
- 下一步行动

目标是帮助用户从“情绪化焦虑”进入“结构化判断”。

---

## 2. 适用场景

适合用于：

- 老板不知道一件事该不该做
- 企业项目推进中出现卡点
- 团队反复争论但没有判断标准
- 客户转化、交付、现金流、内容生产出现问题
- 管理者需要判断某个问题是不是 A 类问题
- 企业想把经营判断沉淀成 AI 可执行的诊断流程
- 咨询、教练、私董会场景中的前置问题澄清

不适合用于：

- 替代法律、财务、医疗、投资等专业判断
- 在缺乏事实的情况下给出确定结论
- 直接替用户做重大商业承诺
- 处理涉密企业资料或客户隐私数据
- 把未经验证的推测当成事实

```text
04_A/B/C 类问题分级

09_不建议立刻做什么
10_风险提醒
11_是否建议进入深度诊断
12_机器可读 JSON
```

## 5. 问题分级标准

### A 类问题

A 类问题通常具备以下特征之一：

- 明显影响收入、现金流或客户信任
- 正在影响关键项目交付
- 会造成团队长期内耗或核心人员流失
- 让老板或负责人持续睡不好、反复焦虑
- 会影响家庭关系、合伙关系或关键合作关系
- 不处理会带来更大损失
- 处在用户的影响圈内，至少有部分可行动空间

A 类问题应优先处理，并进入更深入的诊断和行动设计。

---

### B 类问题

B 类问题通常具备以下特征：

- 有影响，但暂时不致命
- 需要观察更多事实
- 可能影响效率，但未直接影响关键结果
- 用户有一定行动空间，但资源投入需要谨慎
- 适合设定观察指标和小规模试验

B 类问题适合做轻量行动、设定期限、继续观察。

---

### C 类问题

C 类问题通常具备以下特征：

- 主要来自情绪、面子、比较或外部噪音
- 处在关注圈内，用户几乎无法影响
- 即使处理，对收入、客户、团队、家庭关系影响很小
- 投入成本高，但实际收益低
- 容易让团队分心

C 类问题通常不建议投入大量资源，应减少关注或延后处理。

## 6. 影响维度评估

诊断时至少从以下维度判断：

```text
收入影响：高 / 中 / 低 / 未知
现金流影响：高 / 中 / 低 / 未知
客户信任影响：高 / 中 / 低 / 未知
团队协同影响：高 / 中 / 低 / 未知
时间精力消耗：高 / 中 / 低 / 未知
情绪压力影响：高 / 中 / 低 / 未知
家庭与关系影响：高 / 中 / 低 / 未知
可控程度：高 / 中 / 低 / 未知
紧急程度：高 / 中 / 低 / 未知
可逆程度：高 / 中 / 低 / 未知
```

不要只看“急不急”，还要看“值不值得”和“能不能动”。

## 7. 执行流程

### Step 1：复述问题

先用更清楚的语言复述用户的问题，避免用户说的是情绪，AI 诊断的却是另一个问题。

---

### Step 2：拆出事实与判断

把用户描述分成：

- 已确认事实
- 用户感受
- 推测判断
- 需要确认的信息

不要把用户情绪当成客观事实。

---

### Step 3：判断问题类型

问题类型可包括：

- 收入问题
- 成本问题
- 现金流问题
- 客户信任问题
- 产品定位问题
- 内容获客问题
- 销售转化问题
- 团队协同问题
- 交付质量问题
- 老板精力分配问题
- 家庭与关系问题
- 外部环境问题

---

### Step 4：进行 A/B/C 分级

根据影响程度、可控程度、资源投入价值，判断问题等级。

输出时必须说明：

```text
我的初步判断：A / B / C 类问题
判断依据：
不确定因素：
```

---

### Step 5：判断关注圈与影响圈

区分：

- 用户能直接控制的部分
- 用户可以影响但不能完全控制的部分
- 用户只能关注但暂时无法改变的部分

优先把行动放在影响圈和控制圈内。

---

### Step 6：形成 72 小时行动建议

行动建议必须具体、低成本、可执行。

建议格式：

```text
未来 24 小时：
未来 48 小时：
未来 72 小时：
```

不要一上来给复杂战略方案。

---

## 8. 输出质量标准

一份合格的问题诊断输出应满足：

- 能让用户看清问题本质
- 明确区分事实、判断和情绪
- 给出 A/B/C 问题等级
- 能说明为什么这样分级
- 能判断是否属于用户影响圈
- 能提出 72 小时内可执行的行动
- 不制造新的焦虑
- 不给超出信息范围的确定结论
- 不替代专业法律、财务、医疗、投资判断

---

## 9. 风险提醒

生成诊断时必须注意：

- 不编造事实
- 不假装已经掌握用户没有提供的数据
- 不做确定的法律、财务、医疗、投资结论
- 不泄露客户和企业隐私
- 不鼓励高风险冲动决策
- 不把所有问题都夸大成 A 类问题
- 不把情绪安慰伪装成经营建议
- 必要时提醒用户补充关键数据或咨询专业人士

## 10. 示例输入

```text
问题描述：我最近一直在做 AI 工作流和 GitHub 样板库，但悦达传媒官网还没做好，Codex 生成的视觉很难看。我不知道应该继续搞官网，还是先沉淀 Skill。

问题背景：我想扩大影响力，也想未来变现。但现在事情很多，容易乱。

涉及对象：我自己、悦达传媒、悦达智科、潜在客户

当前影响：时间精力、品牌可信度、获客路径、情绪压力

已经做过什么：已经上线 GitHub 公开入口和第一个 Skill 样板。

目前最担心什么：做了很多事，但没有形成真正能带来客户信任和商业转化的资产。

期望结果：知道下一步最值得做什么。

可用资源：ChatGPT、Codex、GitHub、已有公众号和视频号、自己的内容经验。
```

## 11. 示例输出骨架

```text
01_问题复述：
你现在的问题不是“官网要不要做”，而是多个公开资产之间的优先级判断。

02_问题类型判断：
这是一个品牌资产建设与资源分配问题，同时涉及内容获客和商业可信度。

03_A/B/C 类问题分级：
初步判断：A 类问题。

判断依据：
- 它影响你未来的客户信任
- 它影响你对外展示的专业度
- 它影响你后续内容、咨询和服务的承接路径
- 它处在你的影响圈内，可以通过明确顺序推进

04_影响圈判断：
可控部分：GitHub 主页、公开样板库、Skill 内容、官网入口文案
可影响部分：潜在客户是否理解你的价值
不可控部分：别人是否马上认可或付费

05_72 小时行动建议：
24 小时内：完成第二个企业问题诊断 Skill
48 小时内：更新 README 和官网入口
72 小时内：把它改写成一篇公众号文章或直播选题
```
