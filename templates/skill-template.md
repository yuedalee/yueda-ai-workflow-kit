# Agent Skill 通用模板

> Template Version v1.0  
> 用途：用于创建新的 Agent Skill / AI 工作流样板，保证不同 Skill 的结构、边界、质量标准和示例格式保持一致。

---

## 使用说明

创建新 Skill 时，建议复制本模板，并放入对应目录：

```text
skills/{skill-name}/SKILL.md
```

推荐同时创建：

```text
skills/{skill-name}/examples/input.md
skills/{skill-name}/examples/output-outline.md
```

其中：

- `SKILL.md`：说明这个 Skill 是什么、适合什么场景、怎么执行。
- `examples/input.md`：说明用户应该如何输入任务信息。
- `examples/output-outline.md`：说明执行后建议输出成什么结构。

# {Skill 中文名称}

> Public Version v1.0  
> 适用场景：一句话说明这个 Skill 主要解决什么问题。

---

## 1. Skill 定位

本 Skill 用于帮助【目标用户】在【具体场景】中完成【核心任务】。

它不是简单的提示词，而是一套可被 AI 稳定调用的任务说明书。

核心目标：

> 把【一个模糊输入 / 一个业务问题 / 一个内容需求】转化为【可执行、可复用、可协同的结构化输出】。

建议说明：

- 这个 Skill 解决什么问题？
- 主要服务谁？
- 和普通提示词有什么区别？
- 最终要交付什么结果？

---

## 2. 适用场景

适合用于：

- 场景 1
- 场景 2
- 场景 3
- 场景 4
- 场景 5

不适合用于：

- 不适用场景 1
- 不适用场景 2
- 不适用场景 3
- 涉及隐私、机密、违法、违规或高风险判断的场景

---

## 3. 输入要求

使用本 Skill 时，建议提供以下信息：

```text
任务名称：
任务背景：
目标用户：
使用场景：
核心目标：
已有素材：
输出用途：
希望风格：
必须包含：
必须避免：
时间限制：
其他约束：
```

如果用户信息不完整，应先基于已有信息生成可用版本，并明确列出“需要补充确认的信息”。不要频繁追问导致流程中断。

---

## 4. 输出结构

每次执行本 Skill，建议输出以下内容：

```text
01_任务复述
02_关键信息缺口
03_核心判断
04_输出正文 / 方案 / 结构
05_执行步骤
06_质量检查标准
07_风险提醒
08_下一步行动建议
09_可选：发布 / 交付 / 使用说明
10_可选：机器可读 JSON
```

可根据具体 Skill 类型调整，但应尽量保持编号清晰、层级稳定。

---

## 5. 执行流程

### Step 1：识别任务目的

先判断用户真正想完成什么，而不是只照着表面指令输出。

需要区分：

- 用户要的是内容？
- 用户要的是判断？
- 用户要的是方案？
- 用户要的是工具结构？
- 用户要的是可执行清单？

---

### Step 2：拆解输入信息

将用户输入拆分为：

- 已确认事实
- 用户判断
- 用户情绪或偏好
- 缺失信息
- 风险信息

不要把用户的主观判断直接当作事实。

---

### Step 3：选择输出框架

根据任务类型选择合适结构：

```text
内容型任务：标题 / 摘要 / 正文 / 配图 / 发布检查
诊断型任务：问题复述 / 分级判断 / 影响评估 / 行动建议
脚本型任务：钩子 / 主体 / 反转 / 结尾 / 分镜 / 封面
复盘型任务：目标 / 结果 / 差距 / 原因 / 保持 / 停止 / 开始
工作流型任务：输入 / 处理 / 输出 / 工具 / 人工审核 / 风险点
```

---

### Step 4：生成结构化输出

输出要求：

- 先给结论，再展开说明
- 层级清楚，方便复制使用
- 不堆砌专业术语
- 不输出无用废话
- 不编造事实
- 不替代专业高风险判断
- 保留人工审核环节

---

### Step 5：生成下一步行动

每个 Skill 最后都应给出下一步行动建议。

建议格式：

```text
下一步 1：
下一步 2：
下一步 3：
```

如果适合，也可以使用：

```text
24 小时内：
48 小时内：
72 小时内：
```

---

## 6. 输出质量标准

一份合格的 Skill 输出应满足：

- 能解决用户提出的核心问题
- 结构清晰，可直接复制使用
- 输出内容与输入信息一致
- 明确区分事实、判断和建议
- 不编造数据、案例或结论
- 不泄露隐私、密钥或客户资料
- 有明确风险提醒
- 有下一步行动建议
- 对公开版和商业版边界有意识

---

## 7. 风险提醒

生成内容时必须注意：

- 不输出 API Key、Webhook、服务器密码等敏感信息
- 不公开客户隐私和企业内部资料
- 不编造客户案例、数据来源和第三方背书
- 不做确定的法律、财务、医疗、投资结论
- 不鼓励用户进行高风险冲动决策
- 不把内部执行说明直接写进面向客户的正文
- 不把公开样板写成完整商业交付资产
